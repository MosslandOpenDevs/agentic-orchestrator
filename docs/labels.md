# GitHub Labels Guide

This document describes the labels used in the Mossland Agentic Orchestrator workflow.

## Quick Reference

| Label | Purpose | Who Adds It |
|-------|---------|-------------|
| `promote:to-plan` | Promote idea to planning | **Human** |
| `promote:to-dev` | Start development from plan | **Human** |
| `type:idea` | Marks an idea issue | Orchestrator/Human |
| `type:plan` | Marks a planning issue | Orchestrator |
| `status:backlog` | In backlog, awaiting action | Orchestrator |
| `status:planned` | Idea has been planned | Orchestrator |
| `status:in-dev` | Currently in development | Orchestrator |
| `status:done` | Completed | Orchestrator/Human |

## Label Categories

### Type Labels

These indicate what kind of issue it is:

- **`type:idea`** - An idea for a new micro Web3 service
  - Created by: Orchestrator (auto-generated) or Human (manual submission)
  - Next step: Add `promote:to-plan` to start planning

- **`type:plan`** - A detailed planning document
  - Created by: Orchestrator (when idea is promoted)
  - Next step: Add `promote:to-dev` to start development

### Status Labels

These track the current state of an issue:

- **`status:backlog`** - In the backlog, waiting for review/action
  - Default status for new ideas and plans

- **`status:planned`** - Idea has been promoted and a plan was created
  - Applied to ideas after `promote:to-plan` is processed

- **`status:in-dev`** - Currently being developed
  - Applied to plans after `promote:to-dev` is processed

- **`status:done`** - Work is complete
  - Applied when development is finished

### Promotion Labels (Human Action)

**These are the only labels humans need to add:**

- **`promote:to-plan`** - Tell the orchestrator to create a plan from this idea
  - Add this to any `type:idea` issue you want to develop
  - The orchestrator will:
    1. Generate a detailed planning document
    2. Create a new `type:plan` issue
    3. Update the idea with `status:planned`
    4. Remove `promote:to-plan` and add `processed:to-plan`

- **`promote:to-dev`** - Tell the orchestrator to start development
  - Add this to any `type:plan` issue you want to implement
  - The orchestrator will:
    1. Create project scaffold in the repository
    2. Set up directory structure
    3. Commit initial files
    4. Update the plan with `status:in-dev`
    5. Remove `promote:to-dev` and add `processed:to-dev`

### System Labels

These are managed automatically by the orchestrator:

- **`generated:by-orchestrator`** - Auto-generated content
  - Applied to all issues created by the orchestrator
  - Helps distinguish auto-generated from human-created issues

- **`processed:to-plan`** - Plan was generated from this idea
  - Applied after `promote:to-plan` is processed
  - Prevents duplicate processing

- **`processed:to-dev`** - Development was started from this plan
  - Applied after `promote:to-dev` is processed
  - Prevents duplicate processing

## Setting Up Labels

Run this command to create all required labels in your repository:

```bash
ao backlog setup
```

This will create all the labels listed above with appropriate colors and descriptions.

## Common Scenarios

### "I want to promote an idea to planning"

1. Go to the idea issue on GitHub
2. Click "Labels" in the right sidebar
3. Add the `promote:to-plan` label
4. Wait for the next orchestrator run (or run `ao backlog process`)
5. A new plan issue will be created and linked

### "I want to start development on a plan"

1. Go to the plan issue on GitHub
2. Click "Labels" in the right sidebar
3. Add the `promote:to-dev` label
4. Wait for the next orchestrator run (or run `ao backlog process`)
5. Project scaffold will be created in `projects/` directory

### "I want to submit my own idea"

1. Go to Issues → New Issue
2. Select "Idea Submission" template
3. Fill in the required fields
4. Submit (labels will be added automatically)

### "I want to see what's in the backlog"

```bash
ao backlog status
```

Or on GitHub: Filter issues by `label:status:backlog`
